# المراقبة والسجلات — Key2lix

## السجلات (Pino)

- السيرفر يسجّل كل طلب بصيغة JSON: `method`, `url`, `statusCode`, `ms`.
- الأخطاء تُسجّل عبر `logger.error`.
- التحكم بمستوى السجل عبر المتغير البيئي: `LOG_LEVEL=debug` أو `info` أو `warn` أو `error`.

مثال مخرجات السجل:
```json
{"level":30,"time":...,"method":"GET","url":"/","statusCode":200,"ms":12,"msg":"request"}
```

## تتبع الأخطاء (Sentry) — A20

- **سيرفر (Node):** عند تعيين `SENTRY_DSN` في `.env` يتم تفعيل `@sentry/node`. يُبلَّغ تلقائياً عن أخطاء في: حفظ الطلب، حفظ رسالة Contact، جلب المنتجات.
- **عميل (المتصفح):** الصفحات التي تحمّل `common.js` تجلب `GET /api/config` وتحمّل سكربت Sentry للمتصفح عند وجود `sentryDsn`، فتُبلَّغ أخطاء الجافاسكربت في الواجهة إلى نفس المشروع في Sentry.
- للحصول على DSN: أنشئ مشروعاً في [sentry.io](https://sentry.io) وانسخ الـ DSN من إعدادات المشروع (نفس الـ DSN يُستخدم للسيرفر والعميل إن شئت).

بدون `SENTRY_DSN` لا يتم تحميل Sentry (سيرفر أو عميل) ولا يحدث أي اتصال خارجي.

## خطط مستقبلية (للعمل عليها)

- إضافة مسار صحة (health check) مثل `GET /health` يعيد حالة السيرفر وقاعدة البيانات.
- لوحة بسيطة أو صفحة أدمن لعرض آخر الأخطاء أو إحصائيات الطلبات من السجلات.
