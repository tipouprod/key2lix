# التوسع — عقد متعددة و Queue

عقد متعددة و Queue البريد مُطبَّقان اختيارياً في المشروع.

---

## بيئة متعددة العقد (Multi-node)

**الفكرة:** تشغيل أكثر من عملية Node خلف موازن حمل (مثلاً Nginx أو موازن سحابي) لزيادة الإنتاجية والتوفر.

**متطلبات:**

1. **جلسات مشتركة:** لا تكفي الجلسات في الذاكرة لأن كل عملية لها ذاكرتها. الحلول:
   - **SESSION_STORE=db** ✅ — الجلسات في جدول `sessions` (SQLite أو PostgreSQL) عبر `lib/session-store-db.js`. مناسب لعقد متعددة دون Redis.
   - **Redis (مستقبلي):** استخدام `connect-redis` مع `express-session` إذا رغبت بأداء أعلى للجلسات.

2. **ملف التطبيق وقاعدة البيانات:** نفس الكود ونفس قاعدة البيانات (أو قراءة/كتابة منفصلة لـ PostgreSQL كما في [POSTGRES-MIGRATION.md](POSTGRES-MIGRATION.md)).

3. **الموازن:** تكوين Nginx/Caddy أو موازن الحمل السحابي لتمرير الطلبات إلى العقد (round-robin أو least connections).

---

## Queue للعمليات الثقيلة

**الفكرة:** نقل المهام التي تستغرق وقتاً (إرسال بريد، معالجة صور، تقارير ضخمة) إلى قائمة مهام (job queue) حتى لا تحجب طلبات المستخدم.

**خيارات مستقبلية:**

- **Bull + Redis:** قائمة مهام شائعة لـ Node. تحتاج تثبيت Redis وتشغيل worker منفصل (أو داخل نفس العملية).
- **مهام بسيطة دون Redis:** تشغيل عمليات ثقيلة في `setImmediate` أو `queueMicrotask` لتخفيف الحجب فوراً؛ لا يحل مشكلة توزيع العمل على عدة سيرفرات.

**حالات الاستخدام في Key2lix (مستقبلية):**

- إرسال البريد (إشعارات، استعادة كلمة المرور).
- معالجة صور (تحويل WebP، إنشاء أحجام متعددة) بعد الرفع.
- إنشاء تقارير Excel/PDF كبيرة.

حالياً هذه العمليات تُنفَّذ داخل طلب الـ API؛ عند النمو يمكن نقلها إلى Queue دون تغيير واجهات المستخدم.

---

**عقد متعددة (مُطبَّق):** تشغيل بعدة عقد: **pm2 start ecosystem.config.cjs** (يستخدم cluster) أو **npm run start:cluster** (Node cluster). تأكد من **SESSION_STORE=db** في `.env`.

**Queue البريد (مُطبَّق):** ضع **REDIS_URL** (أو REDIS_HOST) و **QUEUE_ENABLED=1** في `.env`؛ مهام البريد تُضاف إلى Bull ويُعالجها worker في نفس العملية. راجع [QUEUE.md](QUEUE.md).
